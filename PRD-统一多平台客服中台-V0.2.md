# 清力技术统一多平台客服中台 PRD

版本：V0.2  
日期：2026-07-20  
适用范围：官网客服页、多平台接入页、统一 Agent、客户库、知识库、导入导出、审核与同步

## 1. 背景

当前已经有一个可用的客户侧网页客服页，以及一个后台管理页，包含客户档案、知识库、审核、导出和数据库查看能力。

下一阶段不再把产品理解成“单个客服网页”，而是升级为一个可交付的**统一多平台客服中台**。  
这个中台需要同时承接：

- 官网直接咨询
- 小红书、抖音、B 站、淘宝/天猫、京东、快手等平台来的咨询
- 统一知识库回答
- 统一客户档案沉淀
- 统一后台运营和人工跟进

## 2. 产品目标

### 2.1 核心目标

1. 客户可从官网直接咨询
2. 多平台咨询统一进入同一个 Agent
3. 客服在一个后台处理不同平台会话
4. 客户资料、知识库、会话记录统一入库
5. AI 回答基于统一知识库，而不是各页面各自回答

### 2.2 成功标准

- 官网入口可正常发起咨询并留资
- 后台可按平台查看消息
- 同一客户 ID 能回填已填写信息
- 知识资料可分类、审核、下载、导出
- Agent 能基于知识库生成答复建议
- 平台接入能力可分层扩展，不影响主站和后台稳定性

## 3. 产品定位

本产品不是简单的网页客服插件，而是一个面向真实业务的客服中台，包含三层：

1. **客户入口层**  
   官网咨询页、留资页、资料展示页

2. **统一接入层**  
   多平台会话接入、同步、格式适配、授权管理

3. **运营后台层**  
   客户库、知识库、审核、导出、平台状态、人工处理

## 4. 平台接入策略

平台接入分成三类：

### 4.1 A 类：可优先深度接入

- 淘宝 / 天猫
- 抖音
- 快手

这类平台更适合做消息、会话、客服工作台接入。

### 4.2 B 类：先做导流或半接入

- 小红书
- B 站
- 京东

这类平台更可能受开放能力、审核或权限限制，适合先做导流、统一收口、人工中转，再逐步深化接入。

### 4.3 C 类：官网自有入口

- 官网网站直接进入咨询

这是最稳定、最可控、最适合沉淀数据的入口。

## 5. 用户角色

| 角色 | 说明 |
| --- | --- |
| 客户 | 浏览、提问、留资、下载资料 |
| 客服 | 查看会话、回复消息、补充档案 |
| 运营 | 审核知识库、管理导入导出、看数据 |
| 管理员 | 管理平台接入、权限、同步、配置 |

## 6. 典型场景

### 6.1 官网咨询

- 客户从官网进入
- 点击咨询
- 输入问题
- Agent 自动回复
- 必要时弹出留资卡片

### 6.2 平台消息统一处理

- 用户在抖音/淘宝/快手等平台发起咨询
- 消息进入统一后台
- 同一 Agent 负责理解与回复
- 客服可人工接管

### 6.3 知识驱动回答

- 客户问技术、产品、应用场景
- Agent 先查知识库
- 命中则直接回复
- 未命中则提示人工或进入知识维护

### 6.4 客户档案沉淀

- 客户填写资料卡
- 写入客户数据库
- 再次出现时自动回填客户自己填写过的内容

## 7. 功能需求

### 7.1 客户入口页

1. 支持输入问题并发送
2. 支持留资弹窗
3. 留资字段包括：称呼、联系方式、希望交流内容、其他
4. 默认全部非必填
5. 提交后显示确认反馈
6. 已留资客户再次打开时，只回填客户自己提供过的内容
7. 官网页面是默认主入口之一

### 7.2 多平台接入页

1. 每个平台一个独立聊天框或会话区
2. 每个平台显示接入状态、同步状态、授权状态
3. 同一 Agent 在所有平台之间共享知识与规则
4. 平台消息统一汇总到同一后台工作台
5. 支持平台断连、重试、延迟同步提醒

### 7.3 统一 Agent

1. 所有消息先经过总控判断
2. 总控判断意图、风险、是否需要人工
3. 标准问题优先走知识问答
4. 信息不足优先走引导
5. 高风险内容优先走合规和人工转接
6. 输出统一的答复建议或结构化交接摘要

### 7.4 客户数据库

1. 支持按客户 ID 保存一份主档案
2. 支持查看、修改、删除、批量选择
3. 支持跟进阶段手动调整
4. 支持风险标签自定义选择
5. 支持导出 Excel、CSV、JSON
6. 支持查看数据库实际内容
7. 支持导入客户表格并预校验

### 7.5 知识库

1. 支持文件上传、文本粘贴、手动录入
2. 支持知识分类预设 + 自定义输入
3. 支持审核队列
4. 支持查看知识文件与文档列表
5. 支持点击文件名直接下载
6. 支持原始资料、结构化条目、可引用版本三层管理
7. 支持知识与客服问答联动

### 7.6 导入导出

1. 支持 Excel / CSV / JSON 导出
2. 支持导出分析透视图
3. 导出时给出可见的下载路径或下载链接
4. 支持客户档案导入
5. 支持导入前预校验
6. 支持按选中项批量导出

### 7.7 审核与同步

1. 知识审核独立于客户审核
2. 单条知识卡片逐条审核
3. 审核区保持与知识库区并列，不互相遮挡
4. 平台同步状态可见
5. 同步失败可重试
6. 支持同步日志查看

## 9. Agent 路由规则

### 9.1 优先级

1. 合规 / 高风险
2. 人工明确诉求
3. 来源识别
4. 标准问答
5. 长文本分析
6. 留资和画像
7. 知识维护
8. 人工转接

### 9.2 路由目标

- 尽量不打断客户
- 尽量不增加跳转
- 平台内能解决的先平台内解决
- 需要留资时再进入留资流程
- 复杂问题再进入人工

## 10. 非功能需求

1. 页面稳定
2. 后台支持长列表滚动
3. 页面模块不能互相遮挡
4. 数据持久化可恢复
5. 配置可修改
6. 导出文件可找到
7. 前后端文案统一
8. 平台接入失败不影响主站可用性

## 11. 分期规划

### Phase 1：核心底座

- 保留现有官网客服页
- 保留现有后台管理页
- 统一客户库和知识库数据结构
- 完成数据库和审核能力

### Phase 2：平台工作台

- 新建多平台接入页
- 先接入可开放能力更强的平台
- 做统一会话列表和消息收口

### Phase 3：平台深度接入

- 补齐平台授权、同步、重试
- 接入消息格式适配
- 完善 AI 自动回复与人工转接

### Phase 4：产品化

- 权限体系
- 操作日志
- 审核流
- 数据看板
- 多角色协作

## 12. 风险与边界

1. 不承诺所有平台都能无缝原生接入
2. 不承诺所有平台都开放完整消息接口
3. 不强行把客户入口和后台工作台合并成一个页面
4. 不在 MVP 阶段一次性做完整 CRM
5. 平台规则变化可能导致接入方式调整

## 13. 结论

这版产品的核心，不是做一个简单网页客服，而是做一个**统一 Agent + 统一数据库 + 多平台接入工作台 + 官网入口** 的客服中台。  
先做底座，再做平台接入，最后做产品化，是最稳的路线。
