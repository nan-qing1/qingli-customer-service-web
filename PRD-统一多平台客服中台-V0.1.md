# 清力技术统一多平台客服中台 PRD

版本：V0.1  
日期：2026-07-20  
基于现有成果：前端客服页、后台工作台、知识库/客户库、统一 Agent 设计

## 1. 背景

当前已具备两类能力：

1. 客户侧网站咨询页
2. 后台管理页，包含客户档案、知识库、审核、导出、数据库查看等

下一阶段的目标不是重做单一客服页面，而是升级成一个可实际交付的**统一多平台客服中台**，把网站、小红书、抖音、B 站、淘宝/天猫、京东、快手等入口统一接入到一个 Agent 和一个数据库体系中。

## 2. 产品目标

### 2.1 核心目标

- 让客户可以从官网直接咨询
- 让不同平台的咨询进入同一客服体系
- 让客服在一个后台内统一处理各平台消息
- 让 AI 基于统一知识库回复
- 让客户资料、消息、知识文档沉淀到统一数据库

### 2.2 成功标准

- 客户入口页可直接留资或发起咨询
- 后台可看到来自不同平台的会话/消息
- 同一客户 ID 的资料可回填、去重、更新
- 知识库可查看、审核、分类、导入导出
- Agent 能依据知识库生成回答建议

## 3. 产品定位

本产品不是单纯的网页客服插件，而是一个：

- **客户咨询入口**
- **多平台接入工作台**
- **统一知识库与客户数据库**
- **AI 辅助回复中枢**

### 3.1 两个主要页面

#### A. 客户入口页

- 官网直接打开
- 客户发起咨询、留资、下载资料
- 作为统一入口之一

#### B. 平台接入页 / 后台工作台

- 接入不同平台的消息与会话
- 每个平台独立聊天框或会话分区
- 背后统一调用同一个 Agent
- 可查看客户资料、知识库建议、跟进状态

## 4. 用户角色

| 角色 | 权限 |
| --- | --- |
| 客户 | 浏览、咨询、留资、提交问题 |
| 客服 | 查看会话、人工回复、完善客户档案 |
| 运营 | 管理知识库、审核资料、查看导出结果 |
| 管理员 | 管理平台接入、数据库、权限、配置 |

## 5. 核心场景

### 5.1 官网咨询

- 客户从官网进入
- 点击咨询
- 输入问题
- Agent 自动回复
- 必要时留资

### 5.2 平台消息接入

- 小红书/抖音/B站/淘宝/京东/快手等消息进入统一后台
- 每个平台保留自己的聊天上下文
- 后台统一查看、统一回复

### 5.3 客户留资

- 客户填写姓名、联系方式、希望交流内容等
- 资料写入客户库
- 同一客户 ID 再次进入时自动回填已填写内容

### 5.4 知识库维护

- 上传文件或粘贴文本
- 自动结构化
- 进入审核队列
- 审核后进入可引用知识库

### 5.5 AI 辅助回复

- Agent 先根据知识库生成建议
- 不确定时提示人工确认
- 复杂/高风险问题进入人工转接

## 6. 信息架构

### 6.1 客户侧

- 首页咨询入口
- 留资弹窗
- 咨询页
- 资料/知识展示

### 6.2 后台侧

- 客户档案
- 会话列表
- 平台接入管理
- 知识库文件与文档
- 版本审核队列
- 导入导出
- 数据库查看
- 平台状态与日志

## 7. 功能需求

### 7.1 客户入口页

1. 支持输入问题并发送
2. 支持留资弹窗
3. 支持姓名、联系方式、交流内容、其他说明
4. 所有字段默认非必填
5. 客户提交后显示确认文案
6. 已留资客户再次打开时优先回填客户自己填写过的内容

### 7.2 平台接入页

1. 支持多个平台分区展示
2. 每个平台有独立会话框
3. 会话内容在统一 Agent 下处理
4. 支持平台授权状态显示
5. 支持接入状态、错误状态、最近同步时间
6. 支持平台消息与客户档案的关联

### 7.3 统一 Agent

1. 所有来源消息先进入总控层
2. 总控层判断风险、意图、上下文
3. 调用知识问答、引导、转接、合规等能力
4. 统一输出回复建议
5. 可支持人工审核后再发送

### 7.4 客户数据库

1. 按客户 ID 存储一份主档案
2. 支持回填、修改、删除、批量选择
3. 支持手动维护跟进阶段
4. 支持风险标签自定义选择
5. 支持导出 Excel / CSV / JSON
6. 支持查看数据库实际内容

### 7.5 知识库

1. 支持上传文件、导入文本、手动录入
2. 支持知识分类预设 + 自定义输入
3. 支持审核队列
4. 支持文件与文档列表查看
5. 支持点击文件名下载
6. 支持内容分层：原始资料、结构化条目、可引用版本
7. 支持知识库与客服问答联动

### 7.6 导入导出

1. 支持 Excel、CSV、JSON 导出
2. 导出时显示下载目录或下载链接
3. 支持导出透视图
4. 支持批量导入客户资料
5. 导入时进行预校验

### 7.7 数据展示与审核

1. 客户档案可单条查看
2. 知识库文件可单条审核
3. 审核区只占对应模块的可视范围
4. 支持滚动浏览长列表
5. 不同区块保持并排而非上下遮挡

## 8. 数据模型建议

### 8.1 统一数据库结构

建议采用一个数据库，分不同数据类型存储：

| 类型 | 用途 |
| --- | --- |
| customer | 客户档案 |
| knowledge | 知识资料 |
| message | 平台会话消息 |
| review | 审核记录 |
| import_export | 导入导出记录 |
| platform | 平台接入配置 |

### 8.2 客户档案字段

- customerId
- name
- contact
- company
- sourcePlatform
- needType
- stage
- risk
- note
- createdAt
- updatedAt

### 8.3 知识资料字段

- knowledgeId
- title
- category
- source
- visibility
- rawContent
- structuredContent
- fileName
- fileUrl
- status
- version
- createdAt

## 9. Agent 路由原则

### 9.1 路由优先级

1. 合规与高风险
2. 明确人工诉求
3. 来源识别
4. 标准问答
5. 长文本分析
6. 留资与画像
7. 知识维护
8. 人工转接

### 9.2 路由目标

- 尽量少打断客户
- 尽量少跳转
- 尽量在原平台完成首轮交流
- 必要时再进入留资页或人工接待

## 10. 非功能需求

1. 页面响应稳定
2. 后台支持长列表滚动
3. 数据保存可恢复
4. 分类、标签、阶段可配置
5. 页面布局不能互相遮挡
6. 前后端文案统一
7. 导出文件可定位下载位置

## 11. 分期规划

### Phase 1

- 保留现有网站前端
- 保留现有后台工作台
- 统一客户库与知识库的基础数据结构

### Phase 2

- 增加平台接入页
- 接入抖音/淘宝/天猫/快手等可开放能力
- 统一会话管理

### Phase 3

- 接入更多平台
- 提升 AI 回复准确率
- 引入更完整的权限、审计、日志和自动化流程

## 12. 范围外

- 不承诺所有平台都能原生评论区无缝接入
- 不承诺所有平台都能直接开放完整消息 API
- 不把前台和后台强行做成一个页面
- 不在 MVP 阶段做复杂 CRM 全量能力

## 13. 备注

本 PRD 以现有前端客服页和后台工作台为基础，目标是把它升级成一个可交付、可扩展、可接入多平台的统一客服产品。
